# /pact.retro — 意图漂移复盘

**触发时机**：每完成 3-5 个功能，或每 2 周。

> 文件读取见 CLAUDE.md § 4。

---

## Step 1  产品意图对比

对照 PAD.md 的产品目标、核心业务主流程、功能类型定义与实际已完成功能，填写偏差表：

| PAD 主流程 / 目标 | 实际已完成功能 | 一致？ | 偏差原因 |
|------------------|----------------|-------|---------|
| [PAD 描述] | [做出来的] | ✅/⚠️ | [如有偏差] |

提问用户：
- 有没有"做出来和当初想要的不一样"的地方？
- 有没有做完才发现不需要的功能？
- 有没有原本计划但现在优先级降低的功能？

## Step 2  产品流与功能堆叠检查

读取最近 3-5 个已完成 PID / contract / verify：

- 是否每个功能都映射到 PAD 核心业务主流程？
- 主流程功能、辅助功能、管理功能、实验功能的比例是否合理？
- 是否出现多个辅助功能推进，而核心主流程没有完成？
- 是否有功能无法说明成功后用户去向？
- 是否有重复定义的空状态、错误提示、权限不足等体验规则？

输出：

```text
产品流状态：稳定 / 有漂移 / 功能堆叠风险
主要问题：[列表]
建议：[更新 PAD / 拆分功能 / 暂停实验功能 / 补齐主流程]
```

## Step 3  流程质量评估

| 步骤 | 质量 | 主要问题 | 改进 |
|------|------|---------|------|
| PID Card 清晰度 | 高/中/低 | | |
| 契约完整性 | 高/中/低 | | |
| build 检查有效性 | 高/中/低 | | |
| 测试覆盖质量 | 高/中/低 | | |

**根因定位**：当功能做出来和预期不符，偏差在哪一层？
- PID Card 不够清晰（意图层）
- 契约遗漏重要场景（契约层）
- AI 实现时跑偏（执行层）
- 验收才发现需求理解错误（需求层）

## Step 4  架构漂移检查

对照 architecture.md 与最近 3-5 个功能的实现记录：

- 是否有功能绕过模块边界？
- 是否出现实体写入归属不清？
- 是否改变状态机但没有更新 architecture.md？
- 是否改变权限判断位置但没有记录？
- 是否命中 ADR 触发条件但没有写入 decisions？

输出架构漂移风险和建议更新项。

## Step 5  技术债回顾

读取 `knowledge/tech-debt.md` 活跃条目：

| ID | 描述 | 登记时间 | 计划偿还 | 复盘评估 |
|----|------|---------|---------|---------|
| TD-XXX | | | | 按时/推迟/已偿还/取消 |

是否有需要新登记或调整优先级的技术债？

## Step 6  可复用经验清理

读取 `knowledge/patterns.md`：
- 删除只对单个功能有效的流水账。
- 合并重复模式。
- 标记已过时或需要重新验证的条目。
- 将稳定模式保留为短句，避免把 patterns 变成长篇文档索引。

## Step 6.5  意图记录复查

若 `.pact/specs/intent.md` 存在：
- 检查"关键环节"表中"必须"环节是否都已在 PAD 中有对应 Step
- 检查"意图变更记录"中是否有未处理的变更（标注影响功能但该功能已 ship 却未反映变更）
- 检查边界声明是否需要更新（如有新的"不做"决策未记录）
- 输出意图记录健康状态：`意图记录：健康 / 有未处理变更 / 需要更新`

无意图记录时跳过本步。

## Step 7  可选代码简化扫描

如项目已安装 code-simplifier，或开发者明确要求，可对本轮改动文件运行简化扫描。
未安装时不自动安装，不阻断 retro 主流程。

## Step 8  范围适配复查

对照 `scope/fitness.md`：
- 实际遇到的问题是否与风险档案预测一致？
- 是否出现评估时未预见的新风险？
- 产品复杂度是否已超出初始评估，需要重新执行 /pact.scope？

## Step 9  生成复盘记录

保存至 `knowledge/decisions/[日期]-retro.md`：

```markdown
# 复盘 — [里程碑]
日期：[日期] | 覆盖功能：[列表]

## 主要偏差
## 产品流与功能堆叠
## 架构漂移
## 根因分析
## 技术债变更
## 下一迭代改进行动
```
