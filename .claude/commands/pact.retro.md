# /pact.retro — 意图漂移复盘

**触发时机**：每完成 3-5 个功能，或每 2 周。

> 文件读取见 CLAUDE.md § 4。

---

## Step 1  产品意图对比

对照 PAD.md 与实际已完成功能，填写偏差表：

| 原始意图（PAD） | 实际实现 | 一致？ | 偏差原因 |
|--------------|---------|-------|---------|
| [PAD 描述] | [做出来的] | ✅/⚠️ | [如有偏差] |

提问用户：
- 有没有"做出来和当初想要的不一样"的地方？
- 有没有做完才发现不需要的功能？
- 有没有原本计划但现在优先级降低的功能？

## Step 2  流程质量评估

| 步骤 | 质量 | 主要问题 | 改进 |
|------|------|---------|------|
| PID Card 清晰度 | 高/中/低 | | |
| 契约完整性 | 高/中/低 | | |
| build 检查有效性 | 高/中/低 | | |
| 测试覆盖质量 | 高/中/低 | | |

**根因定位**：当功能做出来和预期不符，偏差在哪一层？
- PID Card 不够清晰（意图层）
- 契约遗漏重要场景（契约层）
- AI 实现时跑偏（执行层）
- 验收才发现需求理解错误（需求层）

## Step 3  技术债回顾

读取 `knowledge/tech-debt.md` 活跃条目：

| ID | 描述 | 登记时间 | 计划偿还 | 复盘评估 |
|----|------|---------|---------|---------|
| TD-XXX | | | | 按时/推迟/已偿还/取消 |

是否有需要新登记或调整优先级的技术债？

## Step 4  可复用经验清理

读取 `knowledge/patterns.md`：
- 删除只对单个功能有效的流水账。
- 合并重复模式。
- 标记已过时或需要重新验证的条目。
- 将稳定模式保留为短句，避免把 patterns 变成长篇文档索引。

## Step 5  可选代码简化扫描

如项目已安装 code-simplifier，或开发者明确要求，可对本轮改动文件运行简化扫描。
未安装时不自动安装，不阻断 retro 主流程。

## Step 6  范围适配复查

对照 `scope/fitness.md`：
- 实际遇到的问题是否与风险档案预测一致？
- 是否出现评估时未预见的新风险？
- 产品复杂度是否已超出初始评估，需要重新执行 /pact.scope？

## Step 7  生成复盘记录

保存至 `knowledge/decisions/[日期]-retro.md`：

```markdown
# 复盘 — [里程碑]
日期：[日期] | 覆盖功能：[列表]

## 主要偏差
## 根因分析
## 技术债变更
## 下一迭代改进行动
```

```
  需要重新评估范围：是/否
  改进行动：[高优先级列表]
```
