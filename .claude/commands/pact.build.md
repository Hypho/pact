# /pact.build — 计划、实现、检查

> 文件读取见 CLAUDE.md § 4。

---

## 入口检查

`.pact/contracts/[功能名].md` 必须存在。不存在则停止，提示先执行 /pact.contract。

---

## Step 1  生成组件计划

输出以下格式的组件计划：

```
## 组件计划 — [功能名]

### 文件清单
| 文件路径 | 操作 | 说明 |
|---------|------|------|
| src/[路径] | 新建/修改 | [操作范围] |

### 接口变更（如有 API 变更）
[路径、方法、请求/响应格式]

### 实现顺序
1. 数据层（Model / Schema / Migration）
2. 业务逻辑层（Service / Handler）
3. 表现层（Router / Controller / Component）

### 不确定项
[需要人工确认的设计决策，无则写"无"]
```

**⏸️ 强制暂停：展示组件计划，等待人工确认后才进入 Step 2。**
这是 build 流程中唯一的强制暂停点。

---

## Step 2  逐组件实现 + 即时检查

按计划的实现顺序，**逐个组件**完成实现后立即执行四项检查。
不等所有组件完成再统一检查。

**检查 1 — 技术约束合规**（对照 constitution.md）
- 删除操作是否遵守删除策略
- API 响应是否使用统一格式
- 命名是否符合规范
- 是否引入了未登记的第三方库

**检查 2 — 实体定义一致**（对照 PAD.md）
- 字段名是否与 PAD.md 定义一致
- 状态值是否与 PAD.md 状态机一致

**检查 3 — 契约覆盖确认**（对照 contracts/[功能名].md）
- 该组件负责的 FC/NF 条目是否已实现
- 该组件是否越出 PID Card "明确不做"清单（范围蔓延：本组件范围内）

**检查 4 — 运行时边界扫描**
- 并发写入模式（先读后写、计数器）
- 多表操作但无事务保护
- 外部调用但无失败处理
- N+1 查询风险

检查 4 发现高风险特征时：触发边界警告，暂停等待决策（格式见 boundaries.md）。

发现技术债时：
```
📋 技术债：[描述妥协内容] | 触发条件：[描述]
登记到 tech-debt.md？[A]是 [B]否
```

检查通过时：输出单行确认，继续下一个组件。
发现问题时：描述问题 + 对应规则 + 建议修正，等待处理后继续。

---

## Step 3  跨功能一致性扫描

所有组件完成后执行一次，读取所有已完成功能的契约文件：

- 实体字段名在不同功能间是否一致
- 相同类型操作（删除/列表/错误提示）行为是否统一
- FDG 中定义的共享契约是否被遵守
- 当前功能的实现是否触碰了其他已完成功能的"明确不做"边界

输出：
```
✅ 一致性扫描通过 / ⚠️ 发现 N 个问题：[列表]
```

---

## Step 4  更新 state.md

功能阶段字段更新为 `build-complete`。

---

## 会话中断处理

功能较大、当前会话无法完成所有组件时，在完成当前组件并通过检查后记录断点：

```
功能：[名称] | 阶段：build
已完成组件：[列出]
下次继续：实现 [下一个组件名]
```

下次会话执行 /pact.build 时，读取 state.md 从断点继续。

---

## 完成输出

```
✅ build 完成：[功能名]
  实现组件：[N] 个
  检查：全部通过 / 发现并修复 [N] 个问题
  技术债：[N] 条已登记 / 无
  一致性：通过 / 已修复 [N] 个问题
→ 下一步：/pact.verify
```
